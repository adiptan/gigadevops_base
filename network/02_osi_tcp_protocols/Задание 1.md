# Задание 1: Анализ сетевых пакетов с помощью утилиты tcpdump

**Статус:** ✅ Выполнено  
**Дата:** 2026-03-07  
**Платформа:** Linux (Ubuntu)

---

## Описание задания

Изучить утилиту tcpdump для перехвата и анализа сетевого трафика:
- Установить/проверить наличие tcpdump
- Запустить перехват на сетевом интерфейсе
- Сымитировать сетевую активность (ping, HTTP-запросы)
- Проанализировать протоколы и сопоставить с моделью OSI

**Конечный результат:** Понимание перехвата пакетов, определение протоколов и соотнесение с моделью OSI.

### Пункт 2: Определение доступных сетевых интерфейсов

**Команда:**
```bash
ip link show | grep -E '^[0-9]+:' | awk '{print $2}' | sed 's/:$//'
```

**Вывод:**
```
lo
enp0s3
docker0
br-ebd646405e1e
br-81c1d368fcd6
```

**Анализ:**
- `lo` — loopback интерфейс (локальная петля)
- `enp0s3` — основной физический сетевой интерфейс
- `docker0`, `br-*` — виртуальные мосты Docker

**Выбор:** Для перехвата реального трафика используем `enp0s3`

### Пункт 3: Перехват ICMP трафика (ping)

**Команды:**
```bash
# Запуск tcpdump в фоне для перехвата ICMP пакетов
tcpdump -i enp0s3 icmp -n -c 10 &

# Генерация ICMP трафика
ping -c 3 8.8.8.8
```

**Вывод tcpdump:**
```

listening on enp0s3, link-type EN10MB (Ethernet), snapshot length 262144 bytes

13:31:32.787582 IP 192.168.1.35 > 8.8.8.8: ICMP echo request, id 54663, seq 1, length 64
13:31:32.860729 IP 8.8.8.8 > 192.168.1.35: ICMP echo reply, id 54663, seq 1, length 64
13:31:33.789132 IP 192.168.1.35 > 8.8.8.8: ICMP echo request, id 54663, seq 2, length 64
13:31:33.862466 IP 8.8.8.8 > 192.168.1.35: ICMP echo reply, id 54663, seq 2, length 64

```

**Анализ перехваченных пакетов:**
- **ICMP echo request** — запрос ping от клиента (192.168.1.35) к серверу (8.8.8.8)
- **ICMP echo reply** — ответ от сервера клиенту
- **id 54663** — идентификатор сессии ping
- **seq 1, 2, 3** — номера последовательности пакетов
- **length 64** — размер ICMP payload (64 байта)

**Анализ TCP handshake:**

## Соотнесение с моделью OSI

### Модель OSI (7 уровней)

| Уровень | Название | Протокол/Технология | Что видно в tcpdump |
|---------|----------|---------------------|---------------------|
| **7** | Прикладной (Application) | HTTP, DNS | `GET / HTTP/1.1`, `HTTP/1.1 200 OK` |
| **6** | Представления (Presentation) | SSL/TLS, кодировки | — (не видно в plain HTTP) |
| **5** | Сеансовый (Session) | NetBIOS, RPC | — (управление сессиями) |
| **4** | Транспортный (Transport) | TCP, UDP, ICMP | `Flags [S]`, `seq`, `ack`, `win` |
| **3** | Сетевой (Network) | IP, ICMP | `192.168.1.35 > 8.8.8.8`, `ttl`, `id` |
| **2** | Канальный (Data Link) | Ethernet, MAC | `link-type EN10MB`, MAC-адреса |
| **1** | Физический (Physical) | Кабель, радиоволны | — (вне зоны видимости tcpdump) |

### Разбор ICMP пакета по OSI

**Захваченный пакет:**
```
IP 192.168.1.35 > 8.8.8.8: ICMP echo request, id 54663, seq 1, length 64
```

**Уровни:**
- **Уровень 3 (Network):** IP-протокол, адреса источника `192.168.1.35` и назначения `8.8.8.8`
- **Уровень 4 (Transport):** ICMP-протокол, тип `echo request`, идентификатор `54663`
- **Уровень 7 (Application):** Команда `ping` (приложение)

**Вывод:** ICMP работает на уровне 3 (сетевом), но относится к IP-стеку, используется для диагностики.

---

### Разбор TCP/HTTP пакета по OSI

**Захваченный пакет:**
```
192.168.1.35.49522 > 104.18.26.120.80: Flags [P.], seq 1:75, ack 1, win 63, length 74: HTTP
GET / HTTP/1.1
Host: example.com
```

**Уровни:**
- **Уровень 3 (Network):** IP-адреса `192.168.1.35` и `104.18.26.120`
- **Уровень 4 (Transport):** TCP-протокол, порты `49522` (клиент) и `80` (сервер), флаги `[P.]`, `seq`, `ack`
- **Уровень 7 (Application):** HTTP-протокол, метод `GET`, заголовок `Host: example.com`

**Вывод:** HTTP работает на уровне 7 (прикладном), использует TCP на уровне 4, IP на уровне 3.
